# Vibe-Coding 워크플로 (UBG MCP + Angelscript)

> **대상:** ProjectFreeHero 초·중기 개발. 속도 우선, 버그 최소.
> **철학:** 작게 만들고, 바로 돌려보고, 곧바로 커밋. 대화 단계는 최소화.

---

## 0. 왜 CCGS 의식(儀式)을 생략하는가

CCGS 기본 정책(`COLLABORATIVE-DESIGN-PRINCIPLE.md`)은 **질문 → 선택지 → 결정 → 초안 → 승인** 5단계를 거칩니다. 이건 **팀 개발·출시 준비 단계**에 맞춘 안전장치지, 혼자서 프로토타입 만드는 단계엔 과합니다.

**단계별 의식 밀도**:

| 프로젝트 단계 | 권장 의식 밀도 | 이유 |
|---|---|---|
| **지금 (PoC·프로토타이핑)** | 거의 없음 — 바로 코드 | 실험이 목적. 틀려도 되감기 쉬움 |
| 수직 슬라이스 | 중간 — 기능 단위 커밋, 플레이테스트 | 방향 수정이 싸짐 |
| 프로덕션 | 높음 — GDD·리뷰·아키텍처 결정 | 실수 비용 급증 |
| 출시 준비 | 매우 높음 — 전체 의식 | 되돌릴 수 없음 |

**지금은 1번 단계입니다.** 나중에 2번으로 넘어갈 때 신호가 뜨면 그때 의식을 켜면 됩니다 (아래 §7 참고).

## 1. 요청 공식 — 한 줄에 네 가지 담기

```
[파일 경로] + [동사 + 명사 (작은 단위)] + [제약 1개] + [검증법]
```

**예시 (좋음)**:

> `Script/Core/StaminaComponent.as` 만들어줘. `UActorComponent` 상속, MaxStamina/CurrentStamina float, `Drain(float)` / `Regen(float)` 두 함수. 음수 들어오면 무시. 테스트로 Play 시 Log로 현재 값 찍히게 해줘.

**반례 (나쁨)**:

> 스태미나 시스템 만들어줘

나쁜 요청은 Claude가 3번 되묻거나 엉뚱한 걸 만듭니다. 좋은 요청은 즉시 코드로 간다.

### 체크리스트

- [ ] **파일 경로가 있나?** (`Script/...as` 또는 `/Game/BP_...`)
- [ ] **클래스·함수가 하나의 책임을 갖나?** (컴포넌트 하나, 함수 3개 이하가 이상적)
- [ ] **제약 1개 이상** (값 범위, 엣지 케이스, 예외 처리)
- [ ] **"이게 돌아간다"를 어떻게 증명할지** (Log 출력, Play, compile 통과)

## 2. Claude Code에게 맡길 것 vs 직접 할 것

**Claude Code에게 맡기기 좋은 것**:
- `.as` 초안 작성 (반복적 보일러플레이트)
- BP 생성 + 컴포넌트 조합 (`mcp__unreal-handshake__blueprint`)
- BP 그래프 와이어링 (`build_blueprint_graph`)
- 여러 파일에 걸친 동시 변경 (함수 이름 바꾸기 등)
- "컴파일 통과할 때까지 수정해줘" 식의 루프
- 코드 리뷰·버그 원인 찾기

**직접 하는 게 빠른 것**:
- 1–2줄 수정 (VS Code에서 타이핑이 빠름)
- 레벨 에셋 배치·라이팅 (시각 판단 필요)
- BP 그래프 디테일 튜닝 (드래그가 빠름)
- 머티리얼·Niagara 그래프 편집
- Play 버튼 누르고 플레이테스트

**판단 기준**: "Claude에게 설명하는 시간 vs 직접 하는 시간" — 30초 안에 직접 되면 직접 하세요.

## 3. 이터레이션 루프 — 5분 안에 한 사이클

```
0분  요청 입력                          "SomeComponent.as 만들고 BP_X에 붙여줘"
1분  Claude가 .as 작성 + MCP로 BP 조작
2분  VS Code에서 .as 열어서 훑어보기    ← 사람이 내용 파악
3분  UE Editor → Output Log에서 reload 확인 → Play
4분  결과 보고 OK면 git commit, 아니면 고쳐달라고 말함
5분  다음 요청
```

**핵심 규칙**: **한 루프에 한 가지만 바꾼다.** 여러 개 한꺼번에 맡기면 어디서 터진지 모릅니다.

## 4. 버그 최소화하는 습관 5개

### ① 컴파일 통과를 Claude에게 증명시키기

요청 끝에 이거 한 줄 붙이세요:
> "만든 다음 `compile_blueprint`로 컴파일해서 0 errors 인지 확인해줘."

Claude가 MCP로 컴파일하고 에러 있으면 자기가 고쳐옵니다. 사용자가 Play 누르기 전에 한 바퀴 자정.

### ② Play 후 결과를 말로 설명하기

"작동한다/안 된다"만 말하지 말고:
- ✅ "Output Log에 'Stamina: 80/100' 찍힘, Print String도 떴음"
- ❌ "됨" ← Claude가 다음 작업 판단 못 함

### ③ Hot-reload 실패 신호를 무시하지 않기

`.as` 저장했는데 Output Log에:
```
Error: Hot reload failed due to script compile errors. Keeping all old script code.
```
이게 뜨면 **그 이후 Play는 거짓말**입니다 (옛날 코드 돌아감). 바로 고치기.

### ④ `.as` 한 번에 40줄 이상 쓰지 않기

클래스 하나가 40줄 넘어가면:
- 책임 분리 시그널 — 두 컴포넌트로 쪼개기
- Claude가 만든 거라면 "짧게 리팩터해줘" 요청

### ⑤ 커밋은 매 Play 검증 후, 작게, 자주

```
feat(ability): StaminaComponent 추가 (Drain/Regen, 음수 클램프)
```

한 커밋 = 한 기능. 5분 사이클이 끝날 때마다 커밋하는 게 이상적입니다. 나중에 뭐가 터졌을 때 bisect로 즉시 범인 찾음.

## 5. 금지 패턴 (버그 증식기)

| 안티패턴 | 왜 나쁜가 | 대신 |
|---|---|---|
| "전투 시스템 전체 만들어" | Claude가 40개 파일 쏟아냄, 어디서 터진지 모름 | 컴포넌트 1개씩, Play 검증 후 다음 |
| "방금 거 고쳐줘" (맥락 없이) | 어느 파일 어느 줄인지 모호 | "`StaminaComponent.as:27` — Regen이 MaxStamina 초과 허용. 클램프 추가해줘" |
| `.as`와 BP 연결 안 하고 계속 코드만 추가 | 컴파일 통과해도 게임에서 안 불림 | 코드 추가하면 반드시 Play로 호출 경로 확인 |
| 에디터 재시작 없이 플러그인 활성화 변경 | 핸드셰이크 깨짐, MCP 침묵 | `.uproject`·`.uplugin` 건드리면 재시작 |
| `/gdd-create` 같은 긴 스킬을 작은 일에 사용 | 20분 걸릴 일을 3시간 | §7의 "의식 켜야 할 신호" 뜰 때만 |

## 6. 추천 CCGS 스킬/에이전트 사용법

**프로토타입 단계에선 거의 쓰지 마세요.** 쓸 때만 골라서:

| 상황 | 쓸 것 | 쓰지 말 것 |
|---|---|---|
| 작은 기능 구현 | Claude Code 직접 요청 | ❌ 에이전트 호출 |
| 버그 재현 안 됨 | `qa-tester`에게 "재현 스텝 뽑아줘" | |
| 설계 두 방향 중 고민 | `game-designer` 빠른 의견 (2문단) | ❌ GDD 풀문서 |
| 성능 이상함 | `performance-analyst` 프로파일 권고 | |
| 시스템 3개 이상 얽혀서 방향성 모름 | `producer` 정리 요청 | |
| 아키텍처 결정 (DB·네트워킹 기반) | `technical-director` | |

**슬래시 커맨드 중 프로토타입에 유용한 것만**:
- `/project-stage-detect` — 지금 어느 단계인지 재확인 (간헐적)
- `/sprint-status` — 뭐 했었는지 리마인드 (Haiku, 빠름)

**쓰지 마세요**: `/review-all-gdds`, `/consistency-check`, `/gate-check` — 다 늦은 단계용.

## 7. "의식 모드"로 전환해야 하는 신호

아래 중 하나라도 해당되면 프로토타입 모드 끝, CCGS 의식 켜기:

- [ ] `.as` 파일이 10개 넘어감 (일관성 관리 필요)
- [ ] 같은 버그가 두 번 이상 재발 (테스트 프레임워크 도입 시점)
- [ ] 팀원 1명이라도 합류 (공용 계약 필요)
- [ ] 첫 플레이테스터 세션 계획 (기대치 정리 필요)
- [ ] 이걸 상업적으로 낼 거라고 결심 (GDD·ADR·테스트 필요)

그 시점에 `.claude/docs/coordination-rules.md`의 프로토콜을 켜세요.

## 8. TL;DR — 오늘부터 이렇게

1. VS Code, UE Editor, Claude Code 세 창 열기
2. 요청은 **파일경로 + 작은 동사 + 제약 + 검증법** 포맷
3. 한 루프 = 5분 이내, 한 가지만 바꾸기
4. "컴파일 통과 확인해줘"를 요청 끝에 습관화
5. Play 결과는 말로 설명 (성공·실패만이 아니라 뭐가 보였는지)
6. 매 루프 끝에 커밋
7. 40줄 넘는 `.as`는 쪼개기
8. 에이전트·슬래시커맨드는 의식 모드 신호 뜨기 전엔 쓰지 말기

## 부록: 자주 쓸 요청 템플릿

**새 컴포넌트**:
> `Script/Core/<Name>Component.as` 만들어. `UActorComponent` 상속, [프로퍼티 N개], [메서드 N개 — 시그니처+행동]. [제약 — 값 범위·엣지케이스]. 다 된 뒤 `BP_TestDummy`에 붙이고 `compile_blueprint` 결과 보고해줘.

**기존 컴포넌트 확장**:
> `<파일:줄>` 에 [행동] 추가해. [동작 조건]. 추가만 하고 기존 시그니처는 안 건드려. hot-reload 에러 안 나는지 확인해줘.

**BP 조립**:
> `BP_<Name>` 만들고 [컴포넌트 목록] 붙여. `BP_<Name>.DefaultSceneRoot` 아래에 [계층]. Play에서 확인 가능한 [행동]을 EventGraph에 연결해. `compile_blueprint` 0 errors 확인.

**버그 수사**:
> `<파일:줄>` 근처에서 [증상]이 나오는데, Play에서 [재현 스텝] 하면 Output Log에 [로그 내용]이 떠야 하는데 안 떠. 가능한 원인 3개 뽑고 가장 가능성 높은 거부터 고쳐줘.

**리팩터**:
> `<파일>` 이 너무 큼 (현재 [N]줄). [분리 방향 — 예: 상태/행동 분리]으로 N개 파일로 쪼개줘. 공개 API는 유지. 다 된 뒤 `BP_X`가 여전히 컴파일 되는지 확인.
