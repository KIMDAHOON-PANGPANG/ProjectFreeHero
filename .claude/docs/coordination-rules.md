# 에이전트 코디네이션 룰

1. **수직 위임**: 리더십 에이전트는 부서 리드에게, 리드는 스페셜리스트에게 위임합니다.
   복잡한 결정에서 계층을 건너뛰지 마세요.
2. **수평 협의**: 같은 계층의 에이전트는 서로 협의할 수 있지만, 자신의
   도메인 밖에서는 구속력 있는 결정을 내려서는 안 됩니다.
3. **충돌 해결**: 두 에이전트가 의견이 다르면 공통 부모에게 에스컬레이션합니다.
   공통 부모가 없으면, 디자인 충돌은 `creative-director` 에게, 기술 충돌은
   `technical-director` 에게 에스컬레이션합니다.
4. **변경 전파**: 디자인 변경이 여러 도메인에 영향을 미치면 `producer`
   에이전트가 전파를 조율합니다.
5. **일방적 크로스 도메인 변경 금지**: 에이전트는 명시적 위임 없이 자신의
   지정 디렉터리 밖 파일을 수정해서는 안 됩니다.

## 모델 티어 할당

스킬과 에이전트는 태스크 복잡도에 따라 모델 티어에 할당됩니다:

| 티어 | 모델 | 사용 시점 |
|------|-------|-------------|
| **Haiku** | `claude-haiku-4-5-20251001` | 읽기 전용 상태 확인, 포맷팅, 단순 조회 — 창의적 판단 불필요 |
| **Sonnet** | `claude-sonnet-4-6` | 구현, 디자인 저작, 개별 시스템 분석 — 대부분 작업의 기본값 |
| **Opus** | `claude-opus-4-6` | 다중 문서 종합, 고영향 페이즈 게이트 판정, 크로스 시스템 종합 리뷰 |

`model: haiku` 스킬: `/help`, `/sprint-status`, `/story-readiness`, `/scope-check`,
`/project-stage-detect`, `/changelog`, `/patch-notes`, `/onboard`

`model: opus` 스킬: `/review-all-gdds`, `/architecture-review`, `/gate-check`

그 외 모든 스킬은 Sonnet 을 기본값으로 합니다. 새 스킬을 만들 때 읽고 포맷만 한다면 Haiku 할당;
5개 이상 문서를 종합하며 고영향 출력을 낸다면 Opus 할당; 그렇지 않으면 미지정 (Sonnet).

## 서브에이전트 vs 에이전트 팀

본 프로젝트는 두 가지 구분된 다중 에이전트 패턴을 사용합니다:

### 서브에이전트 (현재, 항상 활성)
단일 Claude Code 세션 내에서 `Task` 를 통해 스폰됩니다. 모든 `team-*` 스킬과
오케스트레이션 스킬이 사용합니다. 서브에이전트는 세션의 권한 컨텍스트를 공유하며,
세션 내에서 순차 또는 병렬 실행되고 부모에게 결과를 반환합니다.

**병렬 스폰 시점**: 두 서브에이전트의 입력이 독립적이라면 (어느 쪽도 다른 쪽 출력이
필요하지 않다면), 대기하지 말고 두 Task 호출을 동시에 스폰하세요.
예: `/review-all-gdds` Phase 1 (일관성) 과 Phase 2 (디자인 이론) 은 독립적 — 동시에 스폰.

### 에이전트 팀 (실험적 — 옵트인)
동시에 실행되는 여러 개의 독립적인 Claude Code *세션*. 공유 태스크 목록을 통해
조율됩니다. 각 세션은 자체 컨텍스트 윈도우와 토큰 예산을 가집니다.
`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 환경변수가 필요합니다.

**에이전트 팀 사용 시점**:
- 작업이 같은 파일을 건드리지 않을 여러 하위시스템에 걸쳐 있을 때
- 각 워크스트림이 30분 이상 걸리고 진정한 병렬성의 이득이 있을 때
- 시니어 에이전트 (technical-director, producer) 가 여러 에픽에서 동시에 작업하는
  3명 이상의 스페셜리스트 세션을 조율해야 할 때

**에이전트 팀을 사용하지 말아야 할 때**:
- 한 세션의 출력이 다른 세션의 입력으로 필요할 때 (순차 서브에이전트 사용)
- 단일 세션 컨텍스트에 들어가는 태스크일 때 (서브에이전트 사용)
- 비용이 우려될 때 — 각 팀 멤버가 독립적으로 토큰을 소비

**현재 상태**: 본 프로젝트에서 아직 사용되지 않음. 최초 채택 시 여기에 사용을 문서화.

## 병렬 태스크 프로토콜

오케스트레이션 스킬이 여러 독립 에이전트를 스폰할 때:

1. 결과를 기다리기 전에 모든 독립 Task 호출을 발행
2. 의존 페이즈로 진행하기 전에 모든 결과 수집
3. 어떤 에이전트라도 BLOCKED 이면 즉시 표면화 — 조용히 건너뛰지 말 것
4. 일부 에이전트는 완료되고 다른 에이전트가 블록되면 항상 부분 리포트 생성
