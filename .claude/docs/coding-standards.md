# 코딩 표준

- 모든 게임 코드는 public API 에 문서 주석을 포함해야 합니다
- 모든 시스템은 `docs/architecture/` 에 상응하는 아키텍처 결정 기록을 가져야 합니다
- 게임플레이 값은 데이터 주도(외부 설정)여야 하며 하드코딩해서는 안 됩니다
- 모든 public 메서드는 유닛 테스트 가능해야 합니다 (싱글톤보다 의존성 주입)
- 커밋은 관련 디자인 문서 또는 태스크 ID 를 참조해야 합니다
- **검증 주도 개발**: 게임플레이 시스템을 추가할 때는 테스트를 먼저 작성합니다.
  UI 변경은 스크린샷으로 검증합니다. 작업을 완료로 표시하기 전에 예상 출력과
  실제 출력을 비교합니다. 모든 구현에는 그것이 작동함을 증명할 방법이 있어야 합니다.

# 디자인 문서 표준

- 모든 디자인 문서는 Markdown 을 사용합니다
- 각 메커니즘은 `design/gdd/` 에 전용 문서를 가집니다
- 문서는 다음 8개 필수 섹션을 포함해야 합니다:
  1. **Overview** -- 한 단락 요약
  2. **Player Fantasy** -- 의도된 감정과 경험
  3. **Detailed Rules** -- 모호하지 않은 메커니즘
  4. **Formulas** -- 변수로 정의된 모든 수식
  5. **Edge Cases** -- 처리되는 특수 상황
  6. **Dependencies** -- 다른 시스템 목록
  7. **Tuning Knobs** -- 식별된 설정 가능한 값
  8. **Acceptance Criteria** -- 테스트 가능한 성공 조건
- 밸런스 값은 그 근거가 되는 수식이나 이유를 링크해야 합니다

# 테스팅 표준

## 스토리 타입별 테스트 증거

모든 스토리는 Done 표시 전에 적절한 테스트 증거가 있어야 합니다:

| 스토리 타입 | 요구 증거 | 위치 | 게이트 레벨 |
|---|---|---|---|
| **Logic** (수식, AI, 상태 머신) | 자동 유닛 테스트 — 반드시 통과 | `tests/unit/[system]/` | BLOCKING |
| **Integration** (다중 시스템) | 통합 테스트 또는 문서화된 플레이테스트 | `tests/integration/[system]/` | BLOCKING |
| **Visual/Feel** (애니메이션, VFX, 감각) | 스크린샷 + 리드 사인오프 | `production/qa/evidence/` | ADVISORY |
| **UI** (메뉴, HUD, 화면) | 수동 워크스루 문서 또는 인터랙션 테스트 | `production/qa/evidence/` | ADVISORY |
| **Config/Data** (밸런스 튜닝) | 스모크 체크 통과 | `production/qa/smoke-[date].md` | ADVISORY |

## 자동 테스트 룰

- **네이밍**: 파일은 `[system]_[feature]_test.[ext]`; 함수는 `test_[scenario]_[expected]`
- **결정성**: 테스트는 매번 동일한 결과를 내야 합니다 — 랜덤 시드 없음, 시간 의존 assertion 없음
- **격리**: 각 테스트는 자체 상태를 셋업/티어다운하며, 실행 순서에 의존하면 안 됩니다
- **하드코딩 데이터 금지**: 테스트 픽스쳐는 상수 파일이나 팩토리 함수를 사용하며, 인라인 매직 넘버를 사용하지 않습니다
  (예외: 경계값 테스트 — 정확한 숫자 자체가 요점인 경우)
- **독립성**: 유닛 테스트는 외부 API, 데이터베이스, 파일 I/O 를 호출하지 않습니다 — 의존성 주입 사용

## 자동화하지 말아야 할 것

- 비주얼 충실도 (셰이더 출력, VFX 외관, 애니메이션 커브)
- "감각" 품질 (입력 반응성, 체감 무게감, 타이밍)
- 플랫폼별 렌더링 (헤드리스가 아닌 타겟 하드웨어에서 테스트)
- 전체 게임플레이 세션 (자동화가 아닌 플레이테스팅으로 커버)

## CI/CD 룰

- 자동 테스트 스위트는 main 으로의 모든 푸시 및 모든 PR 에서 실행됩니다
- 테스트가 실패하면 머지 불가 — 테스트는 CI 에서 블로킹 게이트입니다
- CI 를 통과시키기 위해 실패 테스트를 비활성화하거나 건너뛰지 마세요 — 근본 원인을 수정합니다
